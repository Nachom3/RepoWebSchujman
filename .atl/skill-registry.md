## Compact Rules (inject into sub-agent prompts)

### React + Tailwind (apply to ALL frontend work)

**Component architecture**:
- Apply atomic design: atoms (Input, Button, Alert) → molecules (FormField, Card) → organisms (AuthForm) → pages.
- Apply container/presentational pattern: pages hold state + side effects, presentational components are pure render with props.
- One responsibility per file. No prop drilling beyond 2 levels — use context or composition.
- Co-locate component + its variants in same folder when tightly coupled.

**Tailwind v3 best practices** (project uses 3.4.19, NOT v4):
- Define design tokens in `tailwind.config.js` under `theme.extend`: colors (semantic: primary, surface, danger, etc.), borderRadius, fontFamily, spacing extensions.
- Use semantic color names (primary, danger, surface) NOT raw palette names (teal-700, amber-600).
- Reusable component classes: use `@layer components` with `@apply` in CSS — but PREFER React components over CSS classes when possible.
- Class composition: use `clsx` or `classnames` for conditional classes; consider `tailwind-merge` to deduplicate conflicting utilities.
- Use `cva` (class-variance-authority) for component variants if complexity grows.
- Move font loading to `<link>` in index.html (preconnect + stylesheet), NOT `@import url()` in CSS (blocks render).
- Strip arbitrary repeated class strings — they signal a missing component.

**Accessibility**:
- All inputs have associated `<label>` (use `htmlFor` or wrap).
- All interactive elements keyboard-reachable with visible focus ring.
- Color contrast WCAG AA minimum.
- Use semantic HTML (`<main>`, `<section>`, `<button type=...>`, etc.).

**React 19 / Hooks**:
- No `useEffect` for derived state — compute during render.
- Effect dependencies must be exhaustive; never silence eslint-plugin-react-hooks.
- Custom hooks for reusable logic (e.g. `useAuth`, `useApi`).
- Avoid prop drilling — context for cross-cutting concerns (auth, theme).

### Backend Node/Express (apply to ALL backend work)

**Security baseline**:
- No secret fallbacks in code. App MUST fail to start if required secrets are missing.
- bcrypt: min 12 rounds in 2026.
- Server-side input validation ALWAYS (express-validator, joi, or zod). Client-side validation is UX only.
- CORS: explicit `origin` whitelist, never `app.use(cors())` open.
- Rate limiting on auth endpoints (`express-rate-limit`).
- Don't leak whether email exists on login (uniform error message).
- Normalize emails: `.trim().toLowerCase()`.

**Error handling**:
- Async/await everywhere; don't mix with callback APIs. If using sqlite3 callbacks, wrap them in promises.
- Centralized error middleware at end of pipeline.
- Never expose raw error messages or stack traces to client in production.

**Code organization**:
- Routes thin, logic in services/controllers.
- Validation as middleware.
- Auth middleware reused across protected routes.
